## 1. 背景与目标

### 1.1 背景

r-man 作为一个通用 AI Agent，可以处理多种任务，需要与用户进行多轮对话。在对话过程中，用户可能会切换话题，导致之前的上下文信息丢失。为了提高用户体验和系统的连续性，需要设计一个内存系统来保存和管理用户的对话历史和上下文信息。

### 1.2 目标

- **目标 1**: 提供内存管理工具，支持保存和获取对话历史和上下文信息
- **目标 2**: 实现话题切换时的内存自动保存机制
- **目标 3**: 提供灵活的内存检索和管理能力

---

## 2. 业务需求

### BR-001: 话题切换时的内存保存

**描述**: 当用户从一个话题切换到另一个话题时，系统应自动保存前一个话题的对话历史和上下文信息，以便后续可以恢复。

**触发条件**:
- 用户明确表示要切换话题
- 系统检测到话题发生显著变化
- 用户请求开始新的任务

**期望结果**: 前一个话题的完整对话历史和上下文信息被保存到内存系统中，用户可以在后续通过命令恢复或查看。

### BR-002: 内存检索与恢复

**描述**: 用户可以通过命令获取之前保存的话题内存，并继续之前的对话或任务。

**使用场景**:
- 用户需要回到之前的话题继续讨论
- 用户需要参考之前的对话内容
- 用户需要恢复之前未完成的任务

**期望结果**: 系统能够准确检索并返回用户指定的内存内容，用户可以基于此继续对话或任务。

## 3. 功能需求

### FR-001: memory_dump 工具

**描述**: 提供一个内存转储工具，用于将当前对话的上下文信息经过大模型summary后保存到内存系统中。

**功能特性**:
- 自动捕获当前对话的完整历史
- 通过大模型对对话内容进行总结，提取关键信息
- 对总结内容进行向量化处理
- 将向量化结果存储到知识库中
- 支持为内存添加标签和描述
- 保存对话的时间戳和相关元数据
- 支持内存的自动过期管理

**调用时机**:
- 用户明确要求切换话题时
- 系统检测到话题切换时
- 任务完成时自动保存

**参数**:
- `tag` (可选): 内存标签，用于后续检索
- `description` (可选): 内存描述，提供更多上下文信息

**返回值**:
- 内存 ID: 用于后续检索的唯一标识符
- 保存状态: 成功/失败
- 总结内容: 大模型生成的对话总结

---

### FR-002: memory_get 工具

**描述**: 提供一个内存获取工具，用于从内存系统中检索之前保存的对话上下文，支持基于向量化的相似性搜索。

**功能特性**:
- 根据内存 ID 或标签检索内存
- 支持按时间范围、关键词等条件检索
- 支持基于向量化的相似性搜索，根据查询内容与内存的相似度排序
- 返回完整的对话历史和上下文信息
- 支持内存内容的预览和选择

**调用时机**:
- 用户切换新话题时
- 用户明确要求检索历史对话时
- 系统需要参考历史信息辅助当前任务时

**参数**:
- `memory_id` (可选): 内存的唯一标识符
- `tag` (可选): 内存标签
- `query` (可选): 关键词查询或相似性搜索查询
- `limit` (可选): 返回结果数量限制（默认 3）

**返回值**:
- 内存内容: 包含对话历史和上下文信息
- 内存元数据: 标签、描述、时间戳等
- 相似度: 与查询的相似性得分（如果使用相似性搜索）

---

### FR-004: 话题检测与内存管理流程

**描述**: 实现话题检测机制和内存管理流程，在用户切换话题时自动保存前一个话题并检索相关历史对话。

**功能特性**:
- 话题检测（显式和隐式）
- 自动内存保存和检索
- 相似性搜索结果处理

**话题检测机制**:
- **显式检测**: 当用户明确表示"换个话题"、"讨论别的事情"等
- **隐式检测**: 通过语义分析，检测话题的显著变化

**调用流程**:
1. 检测到话题切换事件
2. 调用 memory_dump，将前一个话题的内容 summary 后保存
3. 提取新话题的关键词或核心语义
4. 调用 memory_get 进行相似性搜索
5. 将返回的结果加入到对话上下文中

**技术实现**:
- 在 system prompt 中加入 memory_dump 和 memory_get 的调用时机说明
- 利用向量搜索找到相关历史内存
- 限制返回结果数量（默认最多 3 个），确保响应速度

### FR-003: 内存管理系统

**描述**: 实现一个内存管理系统，用于存储、检索和管理用户的对话内存，包括知识库和向量化处理。

**功能特性**:
- 内存的持久化存储
- 内存的索引和检索
- 内存的过期和清理
- 内存的大小限制和管理
- 知识库管理（存储向量化结果）
- 向量化处理（文本转向量）
- 相似性搜索

**存储机制**:
- 默认采用轻量 SQL：SQLite 存储对话历史、元数据与向量索引（推荐 sqlite_vec 或等价方案）
- 向量化结果与元数据统一落库，避免文件索引与向量索引分离
- 支持 JSON 文件作为兼容迁移输入格式（非默认持久化格式）
- 定期备份和清理

## 4. 非功能需求

### NFR-001: 性能

- 内存保存和检索操作应在 1 秒内完成
- 内存系统应支持至少 100 个内存条目
- 系统应能处理每个内存条目最多 10,000 个 token 的对话历史
- embedding 生成应支持异步处理与批量写入策略，避免高频对话导致阻塞

### NFR-002: 可靠性

- 内存系统应保证数据的一致性和完整性
- 系统崩溃后应能恢复内存数据
- 内存操作应具有原子性，避免部分保存或损坏

### NFR-003: 安全性

- 内存数据应存储在安全的位置，避免未授权访问
- 敏感信息应在内存中适当处理或加密
- 内存清理应彻底，避免数据残留

## 6. 验收标准

### AC-001: memory_dump 工具功能

- 工具能够成功保存当前对话的完整历史
- 工具能够接受并存储标签和描述
- 工具返回唯一的内存 ID
- 工具在用户切换话题时被自动调用

### AC-002: memory_get 工具功能

- 工具能够根据内存 ID 检索内存
- 工具能够根据标签检索内存
- 工具能够返回完整的对话历史和上下文
- 工具支持结果数量限制

### AC-003: 内存管理系统

- 内存数据持久化存储
- 内存系统支持至少 100 个内存条目
- 内存系统能够自动清理过期内存
- 系统崩溃后内存数据可恢复

### AC-004: 性能要求

- 内存保存操作在 1 秒内完成
- 内存检索操作在 1 秒内完成
- 系统能够处理每个内存条目最多 10,000 个 token

## 7. 实现考虑

1. **存储方案**:
   - 使用本地文件系统存储内存数据
   - 每个内存条目存储为单独的 JSON 文件
   - 内存索引存储为一个集中的 JSON 文件

2. **话题检测**:
   - 基于关键词和语义分析检测话题切换
   - 提供用户明确的话题切换命令

3. **内存过期**:
   - 默认内存过期时间为 7 天
   - 支持用户自定义过期时间
   - 定期清理过期内存

4. **集成方案**:
   - 在 ReAct Agent 框架中集成内存管理
   - 在对话处理流程中添加话题检测和内存保存逻辑

5. **用户界面**:
   - 提供简洁的命令接口
   - 支持内存列表查询和管理
   - 提供内存内容预览功能

## 8. 第二阶段落地方案（默认）

### 8.1 架构决策

- 存储引擎：SQLite（轻量 SQL）
- 向量检索：SQLite 向量扩展（如 sqlite_vec）
- embedding 模型：`BAAI/bge-m3`
- 检索策略：向量召回 Top-K + 元数据过滤（tag/time）

### 8.2 配置约定（与 `config/config.yaml` 对齐）

```yaml
memory:
  provider: "sqlite_vec"
  db_path: "./data/memory.db"
  embedding:
    base_url: "${EMBEDDING_BASE_URL}"
    api_key: "${EMBEDDING_API_KEY}"
    model: "BAAI/bge-m3"
  retrieval:
    top_k: 3
    score_floor: 0.0
```

### 8.3 实现说明

- `memory_dump`：写入原始对话、summary 与 embedding 向量
- `memory_get`：仅接收 `memory_id/tag/query/limit`，相似度过滤由服务端 `score_floor` 控制
- 数据迁移：若历史为 JSON 文件，可在启动阶段一次性导入 SQLite
